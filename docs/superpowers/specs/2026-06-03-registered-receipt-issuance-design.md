# 登録済発行の作り直し：窓口での入金記録＋領収書発行

日付: 2026-06-03
対象: 窓口発行 ＞「登録済発行」タブ（`app/ui/issuance_cross_member.py`）

## 背景・課題

「登録済発行」タブの「未発行一覧」が、誰を選んでも常に空になる。

根本原因: 一覧は `get_pending_issuances_for_project_member`（`status=="準備中"` の発行レコードのみ返す）に依存しているが、2026-06-02 の「まとめて発行の簡素化」で旧来の「準備（採番）」ステップが廃止された結果、`Issuance` を作る全経路が作成直後に `mark_as_issued`（または直接 "発行済み"/"支払済み"）するようになり、**"準備中" のまま残るレコードが一件も生成されない**。タブが廃止済みのワークフロー状態を前提にしたまま取り残されている。

## 本来意図していたワークフロー

1. 「まとめて発行」で事業所宛に**請求書**を発行する（status="発行済み"）。
2. 事業所が窓口に来て入金する。
3. その場で**領収書を発行**する（入金記録とセット）。

「登録済発行」は、この 2〜3（窓口での入金→領収書発行）を担う画面であるべき。

## 決定事項

- **進め方**: 「登録済発行」タブを上記ワークフロー向けに作り直す（A案）。「入金管理」タブ（まとめて発行＞）は名簿単位の俯瞰用として現状維持。
- **入金と領収書発行は常にセット**: 窓口で入金を受けたら必ずその場で領収書を発行する。1操作にまとめる。
- **領収書の金額**: 元の請求書の全額・同明細・同宛名をそのまま引き継ぐ（金額編集なし）。

## 設計

### 画面（`IssuanceCrossMemberWidget` を作り直し）

```
[検索：] 事業所名・フリガナ・代表者名
┌─────────────────────────────────────────┐
│ 事業所名 │ 件名 │ 請求書番号 │ 金額 │ 発行日   │  ← 発行済み・未入金の請求書一覧
└─────────────────────────────────────────┘   (1行 ＝ 未入金の請求書1枚)
[入金を記録して領収書を発行]
```

- 検索すると、検索語にマッチする事業所のうち**「発行済み・未入金の請求書」を持つもの**だけを一覧表示する。1行 ＝ 未入金の請求書1枚。
- 行を選択して「入金を記録して領収書を発行」を押すと、既存の入金ダイアログ（入金日＝本日／入金方法／備考）を表示する。OK で以下を1トランザクションで実行:
  1. その請求書に入金を記録し、請求書を「支払済み」化する（`record_payment` 相当）。
  2. 同額・同明細・同宛名で領収書を発行する（status="支払済み"、新規採番）。領収書PDFを生成して開く。
  3. 一覧を再読込し、支払済みになった請求書を一覧から外す。

### 「未入金の請求書」の定義

`doc_type == "invoice"` かつ `status == "発行済み"` の `Issuance`。支払済みになると対象外。領収書・フリー発行分は対象外。

### サービス層

`app/services/issuance_service.py` に新関数を1つ追加する（例: `issue_receipt_for_invoice`）。

- 入力: 元の請求書 `Issuance` の id、入金日、入金額（＝請求書全額）、入金方法、備考、担当者。
- 処理（単一トランザクション）:
  - 元請求書の明細（`IssuanceLine`）・宛名・金額をコピーして領収書 `Issuance` を作成する。採番は領収書として新規に取得（`get_next_doc_number(doc_type="receipt", ...)`）。領収書の status は "支払済み"。
  - `Payment` を作成し、元請求書を "支払済み" に更新する。
- 戻り値: 作成した領収書 `Issuance`（PDF生成に使う）。

これにより「入金済みなのに領収書が存在しない」状態を構造的に防ぐ。

### 検索ロジック

現状の検索（active プロジェクトの `project_members` を走査）を流用しつつ、結果を「未入金の請求書を持つ事業所」に絞る。一覧の単位は請求書（`Issuance`）なので、未入金 invoice から事業所・件名・請求書番号・金額・発行日を引いて表示する。1事業所が複数の未入金請求書を持つ場合は複数行として表示し、個別に領収書を発行する。

## スコープ外（変更しないもの）

- まとめて発行 ＞「入金管理」タブ（`PaymentManagementWidget`）: 名簿単位の俯瞰用として現状維持。
- まとめて発行 ＞「登録データから発行」（`IssuanceFromProjectWidget`）: 一括の請求書/領収書発行として現状維持。
- フリー発行（`IssuanceCounterWidget`）: 変更なし。
- `get_pending_issuances_for_project_member` / `status="準備中"` 関連: 本タブからの利用をやめる。他からの参照がなくなる場合の整理は実装計画で判断する。

## テスト方針

- サービス層: `issue_receipt_for_invoice` のユニットテスト（領収書が元請求書の明細・金額・宛名を引き継ぐ／元請求書が支払済みになる／`Payment` が作られる／領収書番号が新規採番される）。
- UI 層: 検索で未入金請求書を持つ事業所が一覧に出る、支払済みの請求書は出ない、発行後に一覧から消える。
